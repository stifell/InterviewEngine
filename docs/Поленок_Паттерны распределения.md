# Конфигурируемая система автоматической оценки интервью

В основе системы автоматической оценки интервью лежит принцип конфигурации вместо кодирования. Это означает, что правила игры для любой должности определяются в машиночитаемых `YAML`-рубрикаторах. 

Такой подход позволяет гибко настраивать пороговые значения, веса компетенций и сами критерии оценки без единой правки в бэкенде или фронтенде. Для каждой позиции создается отдельный профиль, который задаёт портрет идеального кандидата через набор поведенческих индикаторов с привязкой к шкале BARS (Behaviorally Anchored Rating Scale).

## Базовая структура рубрикатора

В основе системы лежит рубрикатор — `YAML`-файл, в котором для каждой позиции заданы:

*   **Список компетенций** («Владение Go», «Системное мышление», «Ответственность»).
*   **Набор индикаторов** для каждой компетенции с BARS-шкалой (0–5) и порогом приемлемости `acceptableFrom`.
*   **Веса компетенций** (сумма весов = 1).
*   **Правило агрегации**.

## Адаптация под грейды

Чтобы адаптировать систему для трёх грейдов, мы изменяем три параметра:

1.  **`acceptableFrom` каждого индикатора** — определяет, какой минимальный уровень поведения считается проходным для данного грейда.
2.  **Веса компетенций** — для Junior важнее базовые навыки, для Senior — архитектурное мышление и ownership.
3.  **Правило итоговой рекомендации** — можно сделать более гибким, например, разрешить одну слабую компетенцию для Junior (условный проход).

## Паттерны оценки для Go-разработчика
### 1. Junior Go Developer (0–2 года опыта)

>**Цель оценки:** определить, способен ли кандидат решать типовые задачи под руководством старшего разработчика, писать работающий код, понимать базовые концепции Go и не создавать критических проблем.

**Компетенции и их веса**

| Компетенция | Вес |
| :--- | :---: |
| Базовый синтаксис и стандартная библиотека | 0.35 |
| Concurrency (горутины, каналы) | 0.25 |
| Работа с памятью и GC | 0.15 |
| Понимание процессов разработки (Git, CI) | 0.15 |
| Коммуникация и обучаемость | 0.10 |

**Порог приемлемости (`acceptableFrom`)** - 2

**BARS-уровни**
*   **0:** не знает основных конструкций, путает типы.
*   **2:** пишет рабочий код, но может допускать ошибки в обработке ошибок, не использует идиомы.
*   **3:** уверенно пишет чистый код, использует стандартную библиотеку.
*   **4:** знает тонкости (интерфейсы, embedding), пишет тесты.

**Правило итоговой рекомендации**

* **Не прошел:** если хотя бы одна компетенция получила средний балл `< 2` (т.е. есть индикаторы ниже своего `acceptableFrom`).
* **Условно прошел:** все компетенции `≥ 2`, но общий балл `< 2.5` ИЛИ есть одна компетенция со средним баллом `2.0–2.4`. Такого кандидата можно взять, если нет сильных конкурентов.
* **Прошел:** все компетенции `≥ 2.5` и общий балл `≥ 2.5`.

*(В текущей реализации это соответствует `Hire` (условно прошел) и `Strong Hire` (прошел), но пороги сдвинуты вниз).*

### 2. Middle Go Developer (2–5 лет опыта)

> **Цель оценки:** кандидат должен самостоятельно решать задачи средней сложности, проектировать модули, оптимизировать производительность, работать в команде и участвовать в code review.

**Компетенции и веса**

| Компетенция | Вес |
| :--- | :---: |
| Глубокое владение Go (интерфейсы, отражение, профилирование) | 0.30 |
| Concurrency и синхронизация (sync, atomic, select) | 0.25 |
| Работа с памятью, GC, pprof | 0.20 |
| Системное мышление (архитектура микросервисов, API) | 0.15 |
| Коммуникация и наставничество | 0.10 |

**Порог приемлемости (`acceptableFrom`)** - 3

**BARS-уровни**

*   **0:** не знает, как защитить `map`, не использует мьютексы.
*   **2:** знает о `sync.Mutex`, но не различает `RWMutex` и не понимает гонки.
*   **3:** умеет использовать `sync.Map`, понимает детектор гонок, может объяснить блокировку каналов.
*   **4:** знает внутренности планировщика, умеет отлаживать deadlock, использует atomic для простых счётчиков.
*   **5:** может проектировать конкурентные пайплайны с нуля, оптимизировать под нагрузку.

**Правило итоговой рекомендации**

* **Не прошел:** если хотя бы одна компетенция `< 3`.
* **Условно прошел:** все компетенции `≥ 3`, но общий балл `< 3.5` ИЛИ есть одна компетенция ровно `3.0`.
* **Прошел:** все компетенции `≥ 3.5` и общий балл `≥ 3.5`.

*(В текущей системе `Strong Hire` = прошел, `Hire` = условно прошел (если overall ≥ 3.0, но < 3.5). Но в реальности порог для Strong Hire можно поднять до 4.0, а для Hire оставить 3.0).*

### 3. Senior Go Developer / Architect (5+ лет)

> **Цель оценки:** кандидат должен быть экспертом, способным проектировать высоконагруженные системы, принимать архитектурные решения, отвечать за результат команды, обучать других.

**Компетенции и веса**

| Компетенция | Вес |
| :--- | :---: |
| Архитектурное мышление (системный дизайн, trade-offs) | 0.35 |
| Глубокая экспертиза Go (рантайм, планировщик, GC, unsafe) | 0.30 |
| Ownership и ответственность (STAR, лидерство) | 0.20 |
| Производительность и отказоустойчивость | 0.15 |

**Порог приемлемости (`acceptableFrom`)** - 4

**BARS-уровни**

*   **0:** не может предложить архитектуру, путает протоколы.
*   **2:** знает REST, но не сравнивает с gRPC.
*   **3:** обсуждает протоколы и кэширование, но не учитывает CAP.
*   **4:** выбирает протокол на основе требований, обсуждает консистентность, репликацию, шардирование.
*   **5:** проектирует end-to-end систему с учётом отказоустойчивости, геораспределения, мониторинга.

**Правило итоговой рекомендации**

*  **Не прошел:** если хотя бы одна компетенция `< 4`.
*  **Условно прошел:** все компетенции `≥ 4`, но общий балл `< 4.0` ИЛИ есть одна компетенция ровно `4.0`.
*  **Прошел:** все компетенции `≥ 4.5` и общий балл `≥ 4.5`.

*(В текущей системе `Strong Hire` обычно соответствует баллу ≥ 4.0, но для Senior мы поднимаем планку: `Strong Hire` только при ≥ 4.5, а `Hire` при 4.0–4.4).*

## Реализация в YAML-рубрикаторе

Для каждого индикатора задаётся `acceptableFrom`. Например:

```yaml
indicators:
  - id: ind_arch_design
    text: "Выбирает адекватные протоколы и обсуждает trade-offs"
    signals: [termCoverage, answerLength]
    bars:
      0: "Не может предложить архитектуру"
      2: "Знает REST, но не gRPC"
      3: "Обсуждает протоколы и кэширование"
      4: "Выбирает протокол на основе требований, учитывает CAP"
      5: "Проектирует end-to-end систему с отказоустойчивостью"
    acceptableFrom: 4   # для Senior, для Junior было бы 2
```

Агрегатор `ScoreAggregator` использует эти пороги для вычисления `acceptable` каждой компетенции. Итоговая рекомендация вычисляется по правилам, описанным выше (в коде они жёстко зашиты, но при необходимости легко параметризуются через конфиг — например, через отдельные поля `hireThreshold`, `strongHireThreshold` в рубрикаторе).

## Логика работы агрегатора

После того как LLM-судья выставляет баллы по каждому индикатору на основе транскрипта и извлечённых лингвистических признаков, Java-агрегатор вычисляет средний балл по каждой компетенции. 

Затем он сравнивает эти средние с `acceptableFrom` для всех индикаторов, входящих в компетенцию. Если хотя бы один индикатор не достигает своего порога, компетенция помечается как **«неприемлемая»**. Итоговая рекомендация выводится на основе количества неприемлемых компетенций и общего балла, как было описано в каждом паттерне.
