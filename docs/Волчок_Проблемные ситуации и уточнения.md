# Проблемные ситуации и уточнения. Сценарии развертывания и документация

## 1. Проблемные ситуации и уточнения

Проблемные ситуации и уточняющие вопросы используются для формализации содержания структурированного интервью. В системе они задают не свободную беседу с кандидатом, а управляемый оценочный сценарий, в котором каждый вопрос связан с компетенцией, индикатором и шкалой оценки.

Проблемная ситуация - это основной профессиональный контекст, в котором кандидат должен продемонстрировать опыт, способ мышления или уровень владения предметной областью. Такой ситуацией может быть реальный кейс из прошлого проекта, производственный инцидент, конфликт требований, архитектурное решение, технический edge case, диагностическая задача или ситуация выбора профессионального приоритета.

В проекте проблемная ситуация задаётся в YAML-рубрикаторе через поле `mainQuestion`. Уточняющие вопросы оформляются через массив `probes`, при этом каждая проба связана с одним индикатором через поле `indicator`. Индикатор содержит описание проверяемого поведения, список используемых сигналов (`signals`), шкалу BARS и порог приемлемости `acceptableFrom`.

Такая структура позволяет использовать одну и ту же программную логику для разных должностей. При добавлении новой роли меняется YAML-рубрикатор, а не backend-код.

Общая модель проектирования интервью:

```text
Позиция
-> Компетенции
-> Блоки интервью
-> Проблемные ситуации
-> Уточняющие вопросы
-> Индикаторы
-> BARS-шкалы
-> Лингвистические и просодические признаки
-> Итоговая оценка
```

### Типы проблемных ситуаций

Типология проблемных ситуаций может быть общей для разных интервью, а конкретное содержание адаптируется под должность.

| Тип проблемной ситуации | Что проверяется | Пример для технической роли | Пример для аналитической / управленческой роли |
|---|---|---|---|
| Ситуация личной ответственности | Ownership, личный вклад, зона ответственности | Кандидат описывает архитектурный модуль, за который отвечал лично | Кандидат описывает этап проекта или процесса, за который отвечал лично |
| STAR-ситуация из прошлого опыта | Конкретный профессиональный кейс: Situation, Task, Action, Result | Production-инцидент, деградация производительности, сложный технический дефект | Конфликт требований, сложная коммуникация со стейкхолдерами, проблемный релиз |
| Проектировочная ситуация | Системное мышление, trade-offs, выбор решения | Проектирование сервиса, протоколов, кэша, масштабирования | Проектирование бизнес-процесса, схемы интеграции, модели требований |
| Диагностическая ситуация | Поиск причины проблемы по фактам | Анализ роста CPU, GC pressure, latency, ошибок в production | Анализ падения конверсии, расхождения данных, ухудшения процесса |
| Технический или предметный edge case | Глубина знаний в профессиональной области | Go runtime, concurrency, memory, race conditions | SQL, API-контракты, BPMN, требования, ограничения системы |
| Коммуникационная ситуация | Взаимодействие с командой, аргументация, работа с конфликтами | Спор об архитектурном решении, code review, incident post-mortem | Конфликт между бизнесом и разработкой, согласование приоритетов |
| Мотивационная ситуация | Совпадение ожиданий кандидата с ролью | Интерес к архитектуре, инженерной культуре, техническому долгу | Интерес к продукту, доменной области, процессам, команде |

Проблемная ситуация должна быть привязана к профессиональному поведению, которое можно наблюдать и оценить. Поэтому она не должна формулироваться как общий вопрос вида «расскажите о себе» или «что вы знаете о технологии». Корректная проблемная ситуация содержит контекст, ограничения или условия, в которых кандидат должен показать способ принятия решений.

### Роль уточняющих вопросов

Уточняющие вопросы используются для детализации ответа кандидата. Они позволяют получить недостающие элементы: контекст, личную роль, конкретные действия, метрики результата, аргументацию выбора решения и рассмотренные альтернативы.

Уточнение должно быть атомарным: одна проба проверяет один аспект ответа и связывается с одним индикатором. Это важно для автоматической обработки, потому что ответ на пробу затем оценивается по конкретной BARS-шкале.

Некорректный пример уточнения:

 Что вы сделали, почему выбрали это решение, какие метрики улучшились и как команда отреагировала?

Такой вопрос смешивает несколько критериев: действия кандидата, архитектурную аргументацию, результат и коммуникацию.

Корректная декомпозиция:

| Уточнение | Проверяемый аспект |
|---|---|
| Что конкретно сделали именно вы? | Личный вклад |
| Почему выбрали это решение? | Аргументация и trade-offs |
| Какие метрики изменились после внедрения? | Измеримый результат |
| Как решение было принято командой? | Коммуникация и согласование |

### Использование STAR

Метод STAR используется как способ раскрытия проблемной ситуации. Его компоненты не являются самостоятельными блоками интервью, а задают структуру ответа кандидата.

| Компонент STAR | Что раскрывает |
|---|---|
| Situation | В каких условиях возникла проблема |
| Task | Какая задача стояла лично перед кандидатом |
| Action | Какие действия кандидат предпринял |
| Result | Какой результат был получен |

В зависимости от рубрикатора STAR может применяться двумя способами. Первый вариант - разложить компоненты STAR на отдельные уточняющие вопросы и индикаторы. Второй вариант - оценивать полноту STAR-ответа совместно внутри одного индикатора, если требуется оценить целостность профессионального кейса.

Например, в одном профиле STAR может быть разбит на отдельные пробы: ситуация и задача, личные действия, измеримый результат. В другом профиле STAR может использоваться как общий сигнал полноты ответа внутри одного индикатора production-инцидента. В обоих случаях STAR служит не отдельной шкалой, а методикой получения проверяемого ответа.

### Связь с автоматической оценкой

Каждая проблемная ситуация должна быть пригодна для последующей автоматической оценки. Для этого она связывается с индикаторами и признаками ответа.

| Элемент рубрикатора | Назначение |
|---|---|
| `mainQuestion` | задаёт основную проблемную ситуацию |
| `probes` | раскрывают отдельные аспекты ответа |
| `indicator` | связывает уточнение с критерием оценки |
| `signals` | указывает, какие признаки ответа важны |
| `bars` | описывает уровни качества ответа |
| `acceptableFrom` | задаёт минимальный проходной уровень |

После проведения интервью система собирает ответы кандидата по блокам и пробам, извлекает лингвистические признаки (`ownershipRatio`, `hasMetrics`, `termCoverage`, `hedgingDensity`, `answerLength` и другие), при наличии аудио добавляет просодические признаки, после чего передаёт ответ, признаки и BARS-шкалу в LLM-судью.

Итоговая оценка выставляется не за вопрос как таковой, а за индикатор, связанный с этим вопросом. В доменной модели результат оценки индикатора представлен как `IndicatorEvaluation`, который содержит `indicatorId`, балл, признак приемлемости, цитату-доказательство и пояснение.

### Пример применения 

Для профиля **Senior Go Developer / Architect** проблемные ситуации наполняются техническим содержанием: production-инцидент, проектирование сервиса раздачи конфигураций, утечка памяти через slice, concurrent map writes, различие закрытого и nil-канала, давление GC.

Для профиля **System Business Analyst** та же модель наполняется аналитическим содержанием: конфликт требований, описание бизнес-процесса, проектирование интеграционного сценария, работа с API-контрактами, анализ данных, согласование решения со стейкхолдерами.

Таким образом, проблемные ситуации и уточнения являются универсальным механизмом построения гибких интервью. Они позволяют для каждой роли описать собственный набор профессиональных сценариев, но сохранить единую модель оценки: кандидат отвечает на структурированные вопросы, система сопоставляет ответы с индикаторами, извлекает признаки и оценивает их по BARS-шкалам.

## 2. Сценарии развертывания и документация


Проект состоит из нескольких компонентов.

| Компонент | Технологии | Назначение |
|---|---|---|
| Backend | Java 21, Spring Boot 3.5, Spring AI | REST API, загрузка рубрикаторов, запуск транскрипции, оценка интервью, агрегация результатов |
| База данных | H2 для локального режима, PostgreSQL для расширенного режима | Хранение интервью, статусов, транскрипта, результата и кэша оценок |
| LLM-интеграция | Spring AI + Gemini | Оценка ответов по индикаторам и BARS-шкалам |
| Python sidecar | FastAPI, faster-whisper, Praat/parselmouth | Транскрибация аудио/видео и извлечение просодических признаков |
| Frontend | React, Vite, TypeScript, Tailwind, Recharts | Загрузка интервью, отображение скоркарты, радара, тепловой карты и транскрипта |

Backend реализует основной жизненный цикл интервью. Для текстового транскрипта используется endpoint `POST /api/interviews`, для аудио/видео - `POST /api/interviews/from-audio`. Оценка запускается через `POST /api/interviews/{id}/evaluate`, результат возвращается через `GET /api/interviews/{id}/result`.

Также доступны endpoint'ы для получения списка рубрикаторов и деталей конкретного рубрикатора: `GET /api/rubrics` и `GET /api/rubrics/{position}`.

Обработка выполняется асинхронно. Для текстового интервью создаётся запись со статусом `PENDING`, после запуска оценки статус переходит в `RUNNING`, затем в `DONE` или `FAILED`. Для аудио/видео используется дополнительный этап `TRANSCRIBING`: сначала система получает транскрипт и роли спикеров, затем передаёт результат в оценочный конвейер.





### Расширенный сценарий с PostgreSQL

Для режима, приближенного к production, предусмотрен профиль `docker`, в котором backend подключается к PostgreSQL и обращается к sidecar как к отдельному сервису. Конфигурация вынесена в `application-docker.yaml`.

Полный docker-compose в текущей версии проекта находится в разработке, поэтому основным воспроизводимым сценарием остаётся локальный запуск с H2, а PostgreSQL-профиль используется как конфигурационная основа для расширенного развёртывания.

Основные параметры окружения:

| Переменная | Назначение |
|---|---|
| `GEMINI_API_KEY` | ключ доступа к Gemini через Spring AI |
| `SPRING_PROFILES_ACTIVE=docker` | активация docker-профиля |
| `SPRING_DATASOURCE_URL` | JDBC URL PostgreSQL |
| `SPRING_DATASOURCE_USERNAME` | пользователь PostgreSQL |
| `SPRING_DATASOURCE_PASSWORD` | пароль PostgreSQL |
| `TRANSCRIBER_SIDECAR_URL` | адрес Python-сайдкара |
| `WHISPER_MODEL` | модель транскрибации для sidecar |
| `WHISPER_DEVICE` | устройство выполнения: `cpu` или `cuda` |

Миграции базы данных выполняются через Flyway. Первая миграция создаёт таблицу интервью, вторая добавляет поле кэша оценок по индикаторам.

### Документация системы

Документация должна покрывать три уровня работы системы.

| Уровень документации | Содержание |
|---|---|
| Пользовательский | как выбрать позицию, загрузить транскрипт или аудио, запустить оценку и прочитать результат |
| Методический | как устроены проблемные ситуации, уточнения, индикаторы, BARS-шкалы и признаки |
| Технический | как запустить backend, frontend, sidecar, БД, тесты и как настроить переменные окружения |

В проекте выделены документы для методической и технической части: `methodology.md`, `rubric-format.md`, `pipeline.md`, `linguistics-to-bars.md`, `prosody-to-bars.md`. Они описывают формат рубрикатора, шаги конвейера, связь лингвистических и просодических признаков с BARS, а также общую методику оценки структурированных интервью.

### Проверка развертывания

После запуска компонентов корректность системы проверяется через frontend или REST API.

| Шаг | Проверяемое действие |
|---|---|
| 1 | Получить список доступных рубрикаторов через `GET /api/rubrics` |
| 2 | Создать интервью из текстового транскрипта или аудио |
| 3 | Дождаться статуса `DONE` |
| 4 | Получить результат через `GET /api/interviews/{id}/result` |
| 5 | Проверить скоркарту, итоговый балл и рекомендацию |
| 6 | Проверить оценки по индикаторам и цитаты-доказательства |
| 7 | Проверить оценку интервьюера: покрытие вопросов, покрытие проб, соответствие скрипту и тайминг |
| 8 | Проверить отображение результата во frontend: радар, тепловую карту, transcript view и характеристики речи |

В демонстрационном выводе система формирует итоговый балл, рекомендацию, радар по компетенциям, оценку интервьюера, тепловую карту индикаторов, цитаты-доказательства, лингвистические и просодические признаки, а также транскрипт с разделением по блокам и ролям.
